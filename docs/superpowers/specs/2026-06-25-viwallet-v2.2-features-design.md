## 1. Goal

Add the seven most-requested money-management features, each built natively in the v2.1 design language:

| # | Feature | One-line |
|---|---------|----------|
| G | Multi-wallet transfers | Move money between own wallets without double-counting as income/expense |
| A | Savings goals | Track progress toward a target with contributions |
| B | Debt & loan tracking | Money I owe / owed to me, with partial repayments |
| C | Bill reminders | Surface upcoming recurring bills; in-app + local browser reminders |
| E | Search & advanced filters | Full-text + amount/category/date filtering of transactions |
| D | CSV import & export | Bulk import bank statements; export for backup/taxes |
| F | Receipt attachments | Photo receipts on transactions via Supabase Storage |

## 2. Non-Goals

- No changes to the v2.1 visual system, tokens, or animation primitives (reuse them).
- No server-side push infrastructure (VAPID/Web Push) â€” bill reminders are in-app + the local browser Notification API only. Web Push is deferred to v2.3 as an optional stretch.
- No bank-API/Plaid integration â€” import is CSV only.
- No multi-user/shared wallets (single-user scope unchanged).

## 3. Existing Groundwork (reuse, don't recreate)

- `TransactionType` enum already includes `TRANSFER` and `DEBT_LOAN`.
- `Transaction.photos String[]` already exists â€” receipts populate this field.
- `RecurringRule` already powers recurring transactions â€” bill reminders extend it.
- `/api/transactions` GET already supports `walletId/categoryId/tripId/type/from/to/limit/offset` â€” search/filters extend it.

## 4. Data Model Additions

All amounts are integer cents. All tables include `userId` and are RLS-scoped `userId = auth.uid()`. Migrations applied via `prisma db push`; RLS + Storage policies applied via SQL in Supabase (consistent with current deployment).

### 4.1 Transfer (Feature G)
```
Transfer {
  id            String   @id @default(uuid())
  userId        String
  fromWalletId  String
  toWalletId    String
  amount        Int                       // debited from source wallet (source currency)
  toAmount      Int                       // credited to destination (dest currency)
  exchangeRate  Decimal? @db.Decimal(18,8) // null when same currency (rate = 1)
  date          DateTime
  note          String?
  createdAt     DateTime @default(now())
  user          User     @relation(...)
  fromWallet    Wallet   @relation("TransfersFrom", ...)
  toWallet      Wallet   @relation("TransfersTo", ...)
}
```
- POST decrements `fromWallet.currentBalance` by `amount`, increments `toWallet.currentBalance` by `toAmount`.
- Excluded from income/expense reports and budgets (transfers are not spending).
- DELETE reverses both balance effects.

### 4.2 SavingsGoal + GoalContribution (Feature A)
```
SavingsGoal {
  id            String   @id @default(uuid())
  userId        String
  name          String
  icon          String   @default("target")
  color         String   @default("#5DD6A8")
  targetAmount  Int
  currentAmount Int      @default(0)
  currency      String
  deadline      DateTime?
  status        GoalStatus @default(ACTIVE)   // ACTIVE | COMPLETED
  createdAt     DateTime @default(now())
  contributions GoalContribution[]
}
GoalContribution {
  id        String   @id @default(uuid())
  goalId    String
  amount    Int
  walletId  String?                       // optional: deduct from this wallet
  date      DateTime
  note      String?
}
```
- Contribute: increments `currentAmount`; if `walletId` set, decrements that wallet's balance (records as a transfer-like earmark). When `currentAmount >= targetAmount`, set `status = COMPLETED`.

### 4.3 Debt + DebtPayment (Feature B)
```
Debt {
  id              String   @id @default(uuid())
  userId          String
  direction       DebtDirection            // I_OWE | OWED_TO_ME
  counterparty    String
  amount          Int
  remainingAmount Int
  currency        String
  dueDate         DateTime?
  note            String?
  status          DebtStatus @default(OPEN) // OPEN | SETTLED
  createdAt       DateTime @default(now())
  payments        DebtPayment[]
}
DebtPayment {
  id        String   @id @default(uuid())
  debtId    String
  amount    Int
  walletId  String?                       // optional: settle from/to this wallet
  date      DateTime
  note      String?
}
```
- Payment reduces `remainingAmount`; at 0 â†’ `status = SETTLED`. If `walletId` set, adjusts that wallet balance (I_OWE payment debits the wallet; OWED_TO_ME repayment credits it).

### 4.4 RecurringRule extensions (Feature C)
Add fields: `name String?`, `isBill Boolean @default(false)`, `notifyDaysBefore Int @default(1)`. Bills are RecurringRules with `isBill = true`. "Upcoming bills" = bills whose `nextDueDate` falls within `notifyDaysBefore` of today.

## 5. API Surface

| Route | Methods | Notes |
|-------|---------|-------|
| `/api/transfers` | GET, POST | POST validates distinct wallets, applies dual balance update |
| `/api/transfers/[id]` | DELETE | Reverses balances |
| `/api/goals` | GET, POST | |
| `/api/goals/[id]` | GET, PATCH, DELETE | |
| `/api/goals/[id]/contribute` | POST | Increments currentAmount, optional wallet debit, completion check |
| `/api/debts` | GET, POST | |
| `/api/debts/[id]` | GET, PATCH, DELETE | |
| `/api/debts/[id]/payments` | POST | Reduces remainingAmount, optional wallet adjust, settle check |
| `/api/bills/upcoming` | GET | RecurringRules where isBill and due within window |
| `/api/transactions` | GET (extended) | + `q`, `minAmount`, `maxAmount`, `categoryIds` (csv) |
| `/api/transactions/export` | GET | Returns `text/csv` of the user's transactions |
| `/api/transactions/import` | POST | Accepts parsed rows, bulk-creates, updates balances |

All routes auth via `getUser()` and scope by `user.id`, following existing route patterns. Zod-validated bodies.

## 6. UX & Navigation

- **Transfers:** the add-transaction sheet's `PillToggle` gains a third option â€” Expense / Income / **Transfer**. Transfer mode shows From-wallet, To-wallet, amount, and (if currencies differ) an exchange-rate field that computes `toAmount`.
- **Goals:** new `/goals` page (card grid with animated progress rings) + goal detail (contribution history, "Add contribution" sheet).
- **Debts:** new `/debts` page (two sections: "I owe" / "Owed to me", colored) + debt detail (payment history, "Record payment" sheet).
- **Navigation:** bottom nav is capped at 5; add a **"More"** entry that opens a menu linking Goals, Debts, Reports, Settings. Sidebar (desktop) lists all destinations directly.
- **Bill reminders:** dashboard gains an "Upcoming bills" card listing the next due bills with day-count badges; tapping a bill jumps to its recurring rule. On first dashboard visit, request browser Notification permission (dismissible); if granted, schedule local notifications for bills within window.
- **Search & filters:** transactions page gains a search field + a filter sheet (amount range slider, multi-select categories, type, wallet, date range) with active-filter chips.
- **Import/Export:** Settings â†’ "Data" section. Export downloads a CSV immediately. Import opens a wizard: upload â†’ column-map â†’ preview (with validation errors) â†’ confirm â†’ bulk create.
- **Attachments:** the add/edit transaction sheet gains a "Add receipt" control (camera/file). Uploaded images store in Supabase Storage `receipts/{userId}/...`; URLs append to `Transaction.photos`. Transaction detail shows thumbnails with a tap-to-zoom lightbox.

## 7. Supabase Storage (Feature F)

- Bucket `receipts`, private. RLS policy: a user may read/write only objects under the `{auth.uid()}/` path prefix.
- Client uploads via `supabase.storage.from('receipts')`; reads via signed URLs (1-hour expiry) generated server-side or short-lived public URLs.
- Max 5 images/transaction, â‰¤5 MB each, image MIME only â€” validated client-side before upload.

## 8. Non-Functional

- All new pages responsive (375/768/1024/1440) and themed for light/dark via v2.1 tokens.
- All new lists have empty states and skeleton loaders (reusing v2.1 `Skeleton`).
- Balance-affecting operations (transfer, contribution, payment, import) are atomic per request and keep wallet balances correct; DELETE/reverse paths restore balances.
- `npm run build` passes; new logic (balance math, repayment math, CSV parse, filter query building) covered by unit tests.

## 9. Acceptance Criteria

1. A transfer between two wallets moves money correctly, appears in neither income nor expense totals, and reverses cleanly on delete.
2. A savings goal accepts contributions, shows accurate progress, and flips to COMPLETED at/over target.
3. A debt records partial payments, decrements remaining, and flips to SETTLED at zero; "I owe" vs "owed to me" are visually distinct.
4. Bills surface on the dashboard within their notify window; local browser notifications fire when permission is granted.
5. Transactions can be searched by text and filtered by amount range, multiple categories, type, wallet, and date; filters combine correctly.
6. Export produces a valid CSV of all transactions; import maps columns, validates rows, and bulk-creates with correct wallet balances.
7. Receipts upload to user-scoped Storage, attach to a transaction, and display as zoomable thumbnails.
8. Build is green; balance/repayment/CSV/filter logic is unit-tested.
